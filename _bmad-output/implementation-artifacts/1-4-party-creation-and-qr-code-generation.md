# Story 1.4: Party Creation & QR Code Generation

Status: done

## Story

As a host,
I want to create a new karaoke party and receive a QR code and party code,
So that I can invite friends to join my party easily.

## Acceptance Criteria

1. **Given** an authenticated or guest user is on the home screen
   **When** they tap "Start Party"
   **Then** a new session is created in the database with a unique 4-digit party code and status "lobby"
   **And** a QR code is generated that encodes the web landing page URL with the party code
   **And** the host sees both the QR code and the 4-digit party code displayed prominently

2. **Given** a party is being created
   **When** the host selects a party vibe (PartyVibe enum)
   **Then** the vibe persists to the session record

3. **Given** a party is being created
   **When** TV pairing is presented
   **Then** it is shown as optional with a "Skip" option (FR95)

4. **And** the host is automatically added as the first participant in `session_participants`

5. **And** party codes expire after session end and are not reusable (NFR21) — enforced by partial unique index `idx_sessions_active_party_code`. Session end mechanism (`status='ended'`) is implemented in a later story

6. **And** the interaction requires only single taps on targets no smaller than 48x48px (NFR14)
7. **And** no configuration, settings, or preferences are required beyond starting the party (NFR20) — vibe defaults to `general`

    ```typescript
    import { randomInt } from 'node:crypto';

    const CHARS = 'ABCDEFGHJKMNPQRSTUVWXYZ23456789';

    export function generatePartyCode(length = 4): string {
      let code = '';
      for (let i = 0; i < length; i++) {
        code += CHARS[randomInt(CHARS.length)];
      }
      return code;
    }
    ```

    ```dart
    Future<Map<String, dynamic>> createSession({
      String? displayName,
      String? vibe,
      String? venueName,
      String? firebaseToken,
    }) async {
      final headers = <String, String>{'Content-Type': 'application/json'};
      if (firebaseToken != null) {
        headers['Authorization'] = 'Bearer $firebaseToken';
      }
      final body = <String, dynamic>{};
      if (displayName != null) body['displayName'] = displayName;
      if (vibe != null) body['vibe'] = vibe;
      if (venueName != null) body['venueName'] = venueName;

      final response = await http.post(
        Uri.parse('$baseUrl/api/sessions'),
        headers: headers,
        body: jsonEncode(body),
      );
      if (response.statusCode != 201) {
        throw Exception('Failed to create session: ${response.body}');
      }
      final json = jsonDecode(response.body) as Map<String, dynamic>;
      return json['data'] as Map<String, dynamic>;
    }
    ```

      required String serverUrl,

## Dev Notes

### Architecture Compliance

- **Server-authoritative architecture**: Party creation is a server operation. Flutter sends a REST request, server creates the session, Flutter displays the result. No session state created client-side
- **Component boundaries ENFORCED**:
  - `persistence/` is the ONLY layer that imports from `db/` — session-repository.ts and user-repository.ts
  - `services/session-manager.ts` is the ONLY service that orchestrates across layers — it calls persistence and other services
  - `routes/sessions.ts` can call persistence directly (simple auth lookup) AND calls session-manager for lifecycle orchestration
  - `socket-handlers/party-handlers.ts` calls persistence directly for simple operations (vibe update)
- **Provider pattern ENFORCED** (Flutter):
  - ONLY `SocketClient` calls mutation methods on `PartyProvider` and `AuthProvider`
  - Home screen triggers party creation via `SocketClient.createParty()`
  - Lobby screen reads state via `context.watch<PartyProvider>()`

### Party Creation is REST, Not WebSocket

**Chicken-and-egg problem**: The WebSocket auth middleware (from Story 1.3) requires `sessionId` in the handshake. But to create a session, you need to call the server. Therefore:
1. `POST /api/sessions` creates the session via REST (no WebSocket needed)
2. Client receives `sessionId` + `partyCode` in response
3. Client connects WebSocket with the new `sessionId`
4. All subsequent real-time communication uses WebSocket

`POST /api/sessions` is NOT in the architecture's REST API table (which lists only GET endpoints for sessions). This is a justified addition — same pattern as `POST /api/auth/guest` in Story 1.3. Party creation MUST be REST because WebSocket auth requires a sessionId that doesn't exist yet.

The `party:created` event in the Socket.io event catalog is **NOT used for creating parties**. It will be used in Story 1.6 to broadcast to lobby participants that a party session exists.

### Cross-Story Dependencies

- **Story 1.5 (Web Landing Page)**: The QR code URL points to `webLandingUrl` which doesn't exist yet. During development, use a placeholder URL. The QR code will work correctly once the web landing page is deployed
- **Story 1.6 (Party Join Flow)**: Implements `party:joined` WebSocket event that updates lobby participant count. Until then, the lobby shows only the host (count=1)
- **Story 1.7 (Party Start)**: Implements the "START PARTY" button enable logic (3+ participants) and lobby-to-active transition. FR7 (re-display QR during active party) is also addressed in 1.7
- **Story 9.3 (Session Timeline)**: Full session history on home screen. Story 1.4 shows an empty placeholder for authenticated users

### Guest Host User Record Requirement

`sessions.host_user_id` is NOT NULL with FK to `users.id`. Guest hosts MUST have a user row in the database. The `createGuestUser()` function creates a minimal user record with `firebase_uid = NULL`. This also supports future guest-to-account upgrade (Story 9.2) since the user row already exists.

### Party Code Design

- 4 characters from `ABCDEFGHJKMNPQRSTUVWXYZ23456789` (28 chars, excludes ambiguous: 0/O, 1/I/L)
- 28^4 = 614,656 possible codes — sufficient for concurrent sessions
- DB column is `VARCHAR(6)` for future expansion to 5-6 char codes if collision rates require it
- Uniqueness enforced at application level (collision check) AND database level (partial unique index)
- Codes are reusable after session ends (`WHERE status != 'ended'`)
- Use `crypto.randomInt()` NOT `Math.random()` for code generation

### QR Code: Client-Side Generation

QR codes are generated **client-side** in Flutter using `qr_flutter`. Rationale:
- QR encodes a simple URL: `${webLandingUrl}?code=${partyCode}` — no server computation needed
- Instant re-display without network call (FR7: host can re-show QR anytime)
- QR styling (vibe colors) is a Flutter UI concern
- No extra dependency or image payload on server

### Vibe Selector UX Detail

From UX spec: "Micro-preview on tap — Host taps 🎸 → the picker screen background shifts to Rock accent colors for 2 seconds, then resets to current selection." Implementation:

### Anti-Patterns (NEVER DO THESE)

| Wrong | Right |
|-------|-------|
| `Math.random()` for party codes | `crypto.randomInt()` |
| Server-side QR image generation | Client-side `qr_flutter` |
| `party:created` socket event for creation | `POST /api/sessions` REST endpoint |
| Creating session without user record for guest | `createGuestUser()` first, then create session |
| `io.emit('party:vibeChanged', data)` (global) | `socket.to(sessionId).emit(...)` (room-scoped) |
| `import { z } from 'zod'` | `import { z } from 'zod/v4'` |
| `import { fn } from '@/services/session-manager'` | `import { fn } from '../services/session-manager.js'` |
| Widget directly calling `partyProvider.onPartyCreated()` | `SocketClient.createParty()` orchestrates mutations |
| Hardcoded strings in Flutter widgets | All copy in `constants/copy.dart` |
| `Color(0xFF6C63FF)` in a widget | `context.watch<PartyProvider>().vibe.primary` |
| `Padding(padding: EdgeInsets.all(16))` | `Padding(padding: EdgeInsets.all(DJTokens.spaceMd))` |

### Database Schema Reference

```sql
-- Sessions table (from migration 001)
sessions(
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  host_user_id UUID NOT NULL REFERENCES users(id),
  party_code VARCHAR(6) NOT NULL,
  status TEXT NOT NULL DEFAULT 'lobby' CHECK (status IN ('lobby', 'active', 'paused', 'ended')),
  dj_state JSONB,
  event_stream JSONB,
  vibe TEXT,
  venue_name TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  ended_at TIMESTAMPTZ
)
-- Partial unique index: active party codes
CREATE UNIQUE INDEX idx_sessions_active_party_code ON sessions(party_code) WHERE status != 'ended';

-- Session participants (from migration 001)
session_participants(
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL REFERENCES sessions(id),
  user_id UUID REFERENCES users(id),
  guest_name TEXT,
  participation_score INTEGER NOT NULL DEFAULT 0,
  top_award TEXT,
  feedback_score SMALLINT CHECK (feedback_score BETWEEN 1 AND 5),
  joined_at TIMESTAMPTZ NOT NULL DEFAULT now()
)
-- Unique constraint: UNIQUE(session_id, COALESCE(user_id::text, guest_name))
```
